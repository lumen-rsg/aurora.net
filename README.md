# Aurora
*The package manager that doesn't waste your time.*

Aurora is the package manager and bootstrap engine for **lumina**. It is built on a simple premise: package management should be instant, mathematically sound, and bulletproof. We took the battle-tested foundation of the RPM ecosystem, threw away the slow Python wrappers, and built a blazing-fast orchestrator in modern C# compiled to NativeAOT.

---

## Philosophy
We believe in standing on the shoulders of giants, but we aren't afraid to fix their posture.

-   **Native Performance**: Compiled directly to machine code. It starts faster than you can type `au`.
-   **Strict Compatibility**: Fully compatible with standard `.rpm` packages and Fedora repositories. If it installs on Fedora, it installs here. (You can even use Aurora with DNF on the same system)
-   **Delegated Trust**: Aurora handles the complex graph math and parallel network I/O, then hands the payload to the native `rpm` binary to guarantee SELinux contexts, scriptlet execution, and transaction locking.
-   **Intelligent Resolution**: Replaces ancient recursive algorithms with a hashmap solver that handles RPM's rich boolean dependencies without overflowing your stack.

---

## Key Features

### The Core
-   **NativeAOT**: Zero dependencies on the target system. Just a single binary.
-   **Smart Path Normalization**: Automatically resolves UsrMerge conflicts (`/bin` vs `/usr/bin`) during dependency solving.
-   **Parallelism**: Multithreaded downloads using concurrent limits. We saturate your bandwidth, not your mirrors.

### The Repository Engine
-   **Direct SQLite Integration**: No proprietary JSON or YAML databases. Aurora directly downloads and queries standard `repomd.xml` and `primary.sqlite` files generated by `createrepo_c`.
-   **Zero-Overhead Parsing**: Scans and loads metadata for over 80,000 packages in milliseconds.
-   **Fuzzy Matching**: If a dependency fails, Aurora uses Levenshtein distance to suggest the closest available capabilities.

### The Installer
-   **Built-in Bootstrapper**: Includes a highly interactive, UI-driven installation module. Formats disks, handles `chroot` automation, and configures `systemd-boot` or `extlinux` out of the box.

## Usage

Aurora acts as the unified frontend for managing your system state.

### Package Management
**Sync Repositories**
```bash
au sync
```

**Install Packages**
```bash
au install neovim htop git
# Or install a local file directly
au install ./custom-kernel-6.19-1.aarch64.rpm
```

**System Update**
```bash
au update
```

**Remove Packages**
```bash
au remove nano
```

**Audit System Health**
Verifies the cryptographic integrity and permissions of every file on the system.
```bash
au audit
```

### System Bootstrapping
Install lumina to a new mount point (e.g., during live USB installation).
```bash
# Initialize the RPM database skeleton
sudo au --bootstrap /mnt/lumina init

# Sync and install the base system
sudo au --bootstrap /mnt/lumina sync
sudo au --bootstrap /mnt/lumina install lumina-release filesystem setup bash coreutils systemd
```

## Architecture

Aurora is split into three components:

1.  **Aurora.Core**: The brain. Handles SQLite database querying, iterative BFS dependency resolution, and parallel HTTP/Networking logic.
2.  **Aurora.CLI (`au`)**: The user interface. Built with Spectre.Console for clean, interactive, and informative output.
3.  **Aurora.Installer**: The deployment engine. Automates disk mounting, hardware detection, kernel installation, and bootloader provisioning.

---

## Building Aurora

You need the .NET 10 SDK.

```bash
# Publish as a standalone native binary
dotnet publish Aurora.CLI -c Release -r linux-x64 /p:PublishAot=true -o ./out

# The resulting binary is './out/au'
```

## Contributing

We strictly follow the **Lumen Research Group** contribution guidelines.

1.  **Code**: Keep it clean. If your method is longer than your screen, refactor it.
2.  **Performance**: If it allocates unnecessary memory, kill it. We query databases, we do not load them entirely into RAM unless required by the solver.
3.  **Compatibility**: We aim for 100% RPM compatibility. If an `.rpm` installs via `dnf` but fails via `au`, file an issue.
